> **專案全名**：隨手一拍、精準分類：基於多智慧體協同之智慧垃圾分類與幸福減碳地圖

> **最後更新**：2026-06-08  
> **團隊**：Group 14

## 1. 產品概述

### 1.1 產品簡介

EcoCatch 是一款以網頁為基礎的智慧垃圾分類平台，整合多智慧體（Multi-Agent）協同架構，讓使用者透過手機瀏覽器拍照即可辨識垃圾材質、獲得分類指引、導航至最近的回收點，並透過減碳點數激勵持續環保行為。

### 1.2 解決的問題

| 痛點 | 現狀 | EcoCatch 解法 |
|------|------|---------------|
| 垃圾分類困難 | 台灣回收規則複雜，民眾常分錯類 | 拍照即辨識，AI 給出精準分類指引 |
| 找不到回收桶 | 路上公共回收點不明顯、難以搜尋 | GPS 定位 + 地圖導航至最近回收點 |
| 缺乏回收動力 | 正確分類無即時回饋，難以持續 | 減碳點數 + 商家優惠券，正向激勵 |

### 1.3 產品定位

- **平台類型**：響應式網頁應用（Web App），手機優先設計
- **目標市場**：台灣都會區（雙北優先）
- **核心價值**：讓環保回收變得簡單、有趣、有回饋

## 3. 系統架構

### 3.1 多智慧體協同架構 (Multi-Agent Architecture)

本系統由三個核心 AI Agent 協同運作，透過通訊協定即時傳遞資料，共同完成從「影像輸入」到「實體導航」與「點數發放」的完整閉環。

```
┌─────────────────────────────────────────────────────────────────┐
│                        使用者 (瀏覽器)                           │
│                    拍照 → 辨識 → 導航 → 投遞                     │
└──────────┬──────────────────┬──────────────────┬────────────────┘
           │ 影像資料          │ 分類結果+GPS      │ 投遞憑證
           ▼                  ▼                  ▼
  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
  │  Vision Agent   │─→│ Navigation Agent│─→│ Gamification    │
  │  視覺辨識智慧體  │  │ 空間導航智慧體   │  │ Agent           │
  │                 │  │                 │  │ 行為激勵智慧體   │
  │ • 影像分析      │  │ • 最近點搜尋    │  │ • 減碳計算       │
  │ • 材質辨識      │  │ • 路線規劃      │  │ • 點數發放       │
  │ • 分類指引      │  │ • 步行導航      │  │ • 優惠券推送     │
  └─────────────────┘  └─────────────────┘  └─────────────────┘
    資料結構：              資料結構：            資料結構：
    Decision Tree          KD-Tree              Max-Heap
    HashMap                Graph + Min-Heap     HashMap
```

### 3.2 Agent 間通訊流程

```
1. 使用者拍照
       │
       ▼
2. [Vision Agent] 分析影像 → 輸出：{ 類別: "紙容器", 處理步驟: [...] }
       │
       ▼
3. [Navigation Agent] 接收分類 + GPS → 輸出：{ 最近回收點, 步行路線 }
       │
       ▼
4. 使用者前往回收點投遞
       │
       ▼
5. [Gamification Agent] 驗證投遞 → 輸出：{ 點數: +10, 減碳: 0.05kg }
```

## 5. 資料結構設計

### 5.1 Decision Tree（決策樹）

- **用途**：垃圾分類的核心邏輯，將未知物品逐層分類到具體類別
- **存儲方式**：每個節點包含 `{ id, label, question, children[], isLeaf, categoryId }`，以巢狀物件形式儲存
- **為什麼適合**：垃圾分類本身是層層細分的過程（廢棄物 → 可回收/不可回收 → 紙類/塑膠/金屬 → 紙容器/紙箱…），天然符合樹狀結構的特性
- **如何影響功能**：使用者回答問題時，系統從根節點逐層走訪至葉節點，時間複雜度 O(log n)，比線性搜尋所有類別更高效

```
                      [廢棄物]
                     /        \
              [可回收]        [不可回收]
             /   |   \            \
         [紙類] [塑膠] [金屬]    [一般垃圾]
         / \      |      |
    [紙容器][紙箱][PET瓶][鋁罐]
```

### 5.2 HashMap（雜湊表）

- **用途**：儲存每個垃圾類別對應的詳細處理方式與減碳資訊
- **存儲方式**：以 key-value 形式存放，key 為類別 ID（字串），value 為包含處理步驟、回收桶類型、減碳量的物件
- **為什麼適合**：分類完成後需要「根據類別 ID 立即查出處理方式」，這是典型的鍵值查詢場景，HashMap 提供 O(1) 平均時間複雜度
- **如何影響功能**：決策樹走訪完畢得到 categoryId 後，直接用 HashMap.get(categoryId) 取得完整的處理指南，使用者幾乎零等待

### 5.3 KD-Tree（K 維樹）

- **用途**：儲存所有回收點的經緯度座標，用於最近鄰搜尋
- **存儲方式**：二維空間索引樹，交替以經度和緯度作為切分維度
- **為什麼適合**：「找最近的回收點」是典型的最近鄰搜尋問題，KD-Tree 能將暴力搜尋的 O(n) 優化到 O(log n)
- **如何影響功能**：使用者按下導航按鈕後，系統能在毫秒級時間內從數千個回收點中找到最近的幾個

### 5.4 Graph + Priority Queue（圖 + 優先佇列）

- **用途**：路網建模與最短路徑規劃
- **存儲方式**：圖以鄰接表（Adjacency List）儲存，節點為路口/回收點，邊的權重為步行距離；Priority Queue 使用 Min-Heap 實作
- **為什麼適合**：步行路線規劃本質上是圖論的最短路徑問題，Dijkstra 演算法配合 Min-Heap 是經典解法
- **如何影響功能**：找到最近回收點後，Dijkstra 演算法計算最佳步行路線，時間複雜度 O((V+E) log V)

### 5.5 Max-Heap（最大堆積）

- **用途**：維護使用者減碳點數的即時排行榜
- **存儲方式**：以點數為排序依據的完全二元樹，使用陣列實作
- **為什麼適合**：排行榜需要頻繁取出「點數最高的 Top-K 使用者」，Max-Heap 只需 O(K log n)，無需每次完整排序
- **如何影響功能**：每次有使用者投遞成功，O(log n) 更新堆積，排行榜即時反映最新排名

### 5.6 資料結構總覽

| 資料結構 | 所屬 Agent | 核心操作 | 時間複雜度 | 應用場景 |
|----------|-----------|----------|-----------|----------|
| Decision Tree | Vision | 走訪分類 | O(log n) | 垃圾類別判斷 |
| HashMap | Vision / Gamification | 鍵值查詢 | O(1) 平均 | 類別→處理方式對應 |
| KD-Tree | Navigation | 最近鄰搜尋 | O(log n) | 找最近回收點 |
| Graph + Min-Heap | Navigation | 最短路徑 | O((V+E) log V) | 步行路線規劃 |
| Max-Heap | Gamification | Top-K 查詢 | O(K log n) | 點數排行榜 |
